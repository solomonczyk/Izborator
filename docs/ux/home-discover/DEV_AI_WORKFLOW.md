Developer + AI Workflow (v1.0)

1) Роли (кто что делает)
Ты (Dev / Owner)

выбираешь следующую задачу (по списку Task Breakdown)

запускаешь проект / тесты

принимаешь решение “merge / no merge”

ИИ (Architect + UI/UX expert)

даёт план изменения одного файла/компонента за раз

предупреждает о рисках (perf/a11y/архитектура)

даёт короткие acceptance checks

2) Единый цикл на каждую задачу (строго)
Шаг A — Взять задачу

Ты пишешь:

ID задачи (например H-01.3.1)

текущий статус (что уже есть)

что хочешь получить

Шаг B — ИИ отвечает

ИИ выдаёт:

Выводы/риски (2–5 пунктов)

Одно действие (конкретно: какой файл создать/править)

Проверка (как убедиться, что ок)

Шаг C — Ты выполняешь и присылаешь

git diff (кусок)

или содержимое файла

или лог теста

Шаг D — ИИ говорит “OK” и даёт следующий шаг
3) Границы (чтобы не убить архитектуру)
ИИ НЕ делает без запроса

массовые рефакторы

переписывание компонентов

смена архитектуры/паттернов

“давай лучше перепишем на X”

ИИ МОЖЕТ предлагать

улучшения

риски

альтернативы (с ценой/сложностью)

4) Формат “один файл за раз”

Правило:

одна задача = 1–2 файла изменений максимум
(исключение: когда компонент требует отдельного файла + импорт)

Это держит контроль и ускоряет review.

5) Как ты даёшь контекст ИИ (шаблон сообщения)

Копируй и используй каждый раз:

Template

Task: H-XX.X.X

Goal: …

Current: …

Constraints: (не трогать SSR / не менять API / и т.п.)

Output you expect: (какой файл/компонент)

6) QA-минимум на каждый PR

Каждый PR должен иметь:

✅ ручная проверка (описание 1–3 строк)

✅ keyboard check (Tab/Enter)

✅ mobile quick check (chrome devtools)

✅ reduced motion check (если motion затронут)

7) Набор “быстрых вопросов”, которые ИИ обязан закрывать

Для UI задач:

Где риск layout shift?

Где риск a11y (focus/keyboard)?

Где риск perf (mousemove/raf)?

Как это будет на mobile?

Что будет при пустых данных?

8) Как мы избегаем “вайбкодинга”

Запрещаем:

“сделай красиво на глаз”

“добавь анимации ещё”

“пусть будет как Apple”

Делаем:

по спекам

по токенам

по DoD

9) Definition of Done (Workflow)

Workflow считается внедрённым, если:

ты делаешь задачи по ID

изменения маленькие

я даю один шаг

каждый шаг проверяем
